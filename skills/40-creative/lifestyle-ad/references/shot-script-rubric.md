# Shot Script Rubric

Use this reference when writing or reviewing the shot table.

## Required Output Shape

Always produce a table with exactly these columns:

`time | visual | framing/composition | action | sound | subtitle/text | purpose`

## Row Standards

Each row should specify:

- a concrete visible subject
- a camera relationship, such as side profile, low table insert, over-shoulder, hand detail, foreground wipe, hallway follow, or direct creator address
- a physical action or micro-action
- a sound source or silence choice
- text only when it has a job
- the narrative purpose of the shot

## Shot Jobs To Cover

Do not force this order, but cover the necessary jobs for the current concept:

- `status signal`: the desired person-state becomes visible
- `specific friction`: a concrete pressure enters the world
- `composure beat`: the person stays present or in rhythm
- `product gesture`: the product enters through ordinary action
- `proof trace`: the capability is lightly confirmed
- `payoff state`: the user state changes or stabilizes
- `identity close`: the final line names the state or way of living

## Quality Checks

Before asking for user approval, check:

- The first seconds create a viewer question.
- The product appears for a socially natural reason.
- The ad has one core capability, not a feature pile.
- UI or result proof is specific but brief.
- The person is not overacting the pain.
- The camera grammar supports the desired state.
- Text is sparse and judgment-like.
- Copy does not sound like brochure language.
- The concept can be adapted to another category without carrying product-specific assumptions.

## Common Repairs

- If the script feels like a product demo, reduce explanation and add a stronger human state.
- If the script feels generic lifestyle, add a sharper pressure carrier and proof trace.
- If the script feels cheap, reduce motion, reduce text, and make close-ups more selective.
- If the product feels sudden, introduce it through an existing body or environment relationship.
- If proof feels untrustworthy, make the result visible through action, reaction, or a simple object state.

## Approval Prompt

After the table, ask one concise approval question:

`Does this script direction feel right? If yes, I will hand it directly to production.`

If the user approves, do not ask a second confirmation before the production handoff.
