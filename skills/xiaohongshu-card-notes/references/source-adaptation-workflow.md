# Source Adaptation Workflow

This reference explains how to convert source material into Xiaohongshu card pages.

## 1. Find the spine

Before planning pages, identify:

- the main conclusion
- the main problem
- the strongest proof
- the final takeaway

If there is no clear spine, do not start page writing yet.

## 2. Decide the sequence

Most card notes work with one of these sequences:

### Sequence A: Problem -> Insight -> Proof -> Conclusion

Use when the source is judgment-led.

### Sequence B: Hook -> Framework -> Case -> Result

Use when the source is explaining a workflow or method.

### Sequence C: Hook -> Pain -> Capability Matrix -> Proof

Use when the source is product- or tool-led.

### Sequence D: Image Hook -> Judgment -> Dense Explanation -> Proof

Use when the note needs stronger editorial force and higher screenshot value.

## 3. Map source blocks to page roles

### If adapting from HTML

Map:

- hero -> cover
- pain section -> problem cards
- workflow section -> framework or capability cards
- proof section -> proof cards
- result table -> dedicated proof page or multi-page table set

Drop:

- navbar
- footer
- decorative filler
- repetitive CTA blocks

### If adapting from spoken script

Map:

- strongest opening line -> cover hook
- repeated pain statements -> one or two problem cards
- explanation -> framework cards
- metrics / examples -> proof cards
- closing judgment -> final card

Remove:

- spoken transitions
- repeated rhetorical setup
- filler phrases

## 4. Compress each page

For each page ask:

- what is the only thing this page must communicate?
- can this be read in a few seconds?
- is there one stronger line hidden inside this paragraph?

If yes, extract that line and make it the headline.

Then decide:

- is this page a `light card`
- or a `dense memo page`

Choose dense memo when:

- the value lies in the reasoning
- the user has a non-obvious explanation
- the page should be worth saving as a screenshot by itself

## 5. Decide whether proof should be textual or visual

Use textual proof when:

- the point is conceptual
- the numbers are few

Use visual proof when:

- there is a real table
- there are shortlist fields
- the screenshot itself increases trust
- a strong image can create pause before a dense page

## 6. Write design-aware copy

Do not write like a web article.

Write so each page can be laid out cleanly:

- short headline
- short support copy
- explicit data labels when needed

## Default output pattern

For a full note, default to:

1. cover
2. pain
3. hidden problem
4. dense explanation
5. capability matrix
6. proof
7. proof continuation or case
8. closing judgment

Use fewer pages if the source is small.
